## 3.1 実装スコープの初期ゴール

### 1. この repo の最初の到達点はどこか
**回答 / 決定**  
`collector + raw/normalized + Pandera gate + DuckDB quality audit + minimal derived audit + Nautilus export + hftbacktest export` まで。

**理由**  
- collector だけで止めると、取得したデータが backtest に本当に使えるか検証できない
- paper / tiny-live まで入れると、署名・運用・reconciliation でスコープが広がりすぎる
- いま最も重要なのは、**データ取得の真実源**と**execution-aware backtest の入口**を固めること

**実装影響**  
- Phase 1 では `execution/live/` はスタブまたは未着手でよい
- 必須成果物は `normalized/` と `exports/nautilus/`, `exports/hftbacktest/`

**状態**  
承認

## 4. データ基盤で不足している情報

### 11. `l2_book` の normalized 表現を何にするか
**回答 / 決定**  
**2テーブル構成**にする。

1. `book_events`
   - 1 message = 1 row
   - message metadata を持つ
2. `book_levels`
   - 1 level = 1 row
   - `book_event_id` にぶら下げる

**book_events の主列**
- `book_event_id`
- `capture_session_id`
- `reconnect_epoch`
- `book_epoch`
- `symbol`
- `exchange_ts`
- `recv_ts`
- `event_kind` (`snapshot` / `delta` / `recovery_snapshot`)
- `source`
- `raw_msg_hash`
- `n_bids`
- `n_asks`

**book_levels の主列**
- `book_event_id`
- `side`
- `level_idx`
- `price`
- `size`

**理由**  
- 1 message = 配列保持だと audit と export が重い
- 1 level = 1 row なら DuckDB 集計、impact curve、Parquet、Nautilus export がやりやすい
- snapshot / delta 両対応の表現を先に作っておける

**実装影響**  
- raw payload は raw_ws に残す
- normalized は exploded rows を正とする
- Hyperliquid の現行 `l2Book` semantics が snapshot 的でも、schema は delta も受けられる

**状態**  
承認

### 12. channel ごとの primary key / dedup key
**回答 / 決定**  

#### 共通方針
- **primary key** は local surrogate
- **dedup key** は canonical hash（exchange-native id があれば優先）

#### `trades`
- primary: `trade_event_id`
- dedup:
  - native trade id があればそれ
  - なければ `hash(symbol, exchange_ts, px, sz, side, raw_id_if_any)`

#### `l2_book`
- primary: `book_event_id`
- dedup:
  - `hash(symbol, book_epoch, event_kind, canonicalized levels payload)`

#### `active_asset_ctx`
- primary: `asset_ctx_event_id`
- dedup:
  - `hash(symbol, exchange_ts, mark_px, oracle_px, funding_rate, open_interest, mid_px)`

#### `user_fills`
- primary: `fill_event_id`
- dedup:
  - native fill hash or `hash(user, oid, px, sz, exchange_ts)`

#### `order_updates`
- primary: `order_update_event_id`
- dedup:
  - `hash(user, oid_or_cloid, status, exchange_ts, raw_payload_subset)`

**理由**  
- downstream は local surrogate が扱いやすい
- dedup は exchange-native identity がない channel もあるので hash ベースが必要
- reconnect 後の重複 snapshot を落とせる

**実装影響**  
- canonical serializer が必要
- `xxhash64` or `blake2b` を統一利用する

**状態**  
承認

### 13. `capture_session_id` の採番規則
**回答 / 決定**  
**UUIDv7 を collector プロセス起動単位で採番**する。

`reconnect_epoch` は別 integer で増やす。

**理由**

- reconnect と日次 partition を混ぜない方が追跡しやすい

**実装影響**  
- `capture_session_id` は不変
- reconnect のたびに `reconnect_epoch += 1`

**状態**  
承認

### 14. `book_epoch` の意味と切り替え条件
**回答 / 決定**  
`book_epoch` は **order book continuity が保証された連続区間**を表す。  
次の時に increment する。

- 初期 snapshot 取得時
- reconnect 後の再同期 snapshot 受信時
- manual gap repair 後に continuity を切る時
- source 切替（例: official source → fallback source）時

**増やさない**
- 正常連続中の通常 message
- 日次ローテーションだけ

**理由**  
- book continuity を analytics から明示的に扱える
- non-recoverable gap を跨いだ replay を禁止しやすい

**実装影響**  
- `book_epoch` を `book_events` に持つ
- derived で continuity-aware 集計が可能になる

**状態**  
承認

### 15. recoverable gap / non-recoverable gap の閾値
**回答 / 決定**  
保守的に定義する。

#### `l2_book`
- **disconnect / subscription reset / continuity 不明**が起きた時点で  
  **non-recoverable**
- 新 `book_epoch` を開始
- 過去 epoch と後続 epoch は stitching しない

#### `trades`
- recoverable:
  - gap <= 60s
  - かつ後追い backfill source で contiguous recovery できる
- それ以外は non-recoverable

#### `active_asset_ctx`
- recoverable:
  - gap <= 120s
  - fresh snapshot 取得成功
- それ以外は degraded interval 扱い

#### `user_*`
- recoverable:
  - reconnect 後に official info / open orders / fills で整合再構築できる
- 失敗時は non-recoverable

**理由**  
- book continuity は最も厳しく扱うべき
- asset_ctx は state channel なので snapshot 再取得で復帰可能
- trades は backfill source の有無で扱いが変わる

**実装影響**  
- channel ごとの gap policy を config 化
- `non_recoverable_gap_flag` を normalized metadata に残す

**状態**  
承認

**回答 / 決定**  
優先順は次。

1. **WS**
2. **Info endpoint**（bootstrap / short gap repair）
3. **official S3**（official backfill / sanity check）

**実装影響**

- default では Tardis 依存なし
- Tardis を入れる場合も primary source にはしない

### 22. `maker_fill_proxy` / `expected_mfe_conditional` / `edge_to_cost` の初期 proxy
**回答 / 決定**  

#### `maker_fill_proxy`
- empirical fill-rate table を採用
- 条件軸:
  - lane
  - symbol bucket
  - spread bucket
  - funding-distance bucket
  - imbalance bucket
- fallback は hftbacktest の base queue band

#### `expected_mfe_conditional`
- out-of-sample rolling estimate
- 条件軸:
  - regime
  - lane
  - liquidity bucket
  - score bucket
- horizon は **次の 6 x 5m bars**
- conservative estimate は
  - historical median MFE
  - もしくは shrinkage 後の lower quantile

#### `edge_to_cost`
- `expected_mfe_conditional / expected_total_cost_bps`

**理由**  
- いきなり ML は不要
- proxy でも conditional estimate を持たないと L3 gate が実装不能

**実装影響**

## 7. 執行・リスク・live path

### 31. `client_order_id` の deterministic format
**回答 / 決定**  
**内部の `intent_id` と、Hyperliquid に送る `cloid` を分ける。**

#### `intent_id`
人間可読:
`{run_id}-{symbol}-{lane}-{side}-{intent_kind}-{seq}`

#### `cloid`
Hyperliquid 仕様準拠の **16-byte hex string**
生成規則:
- hash input:
  `env | wallet_role | symbol | lane | side | intent_kind | ts_ms_bucket | seq`
- hash:
  `blake2b(digest_size=16)` → hex

**理由**  
- human-readable を external client_order_id に使えない
- deterministic かつ再生成可能にしたい

**実装影響**  
- SQLite では `intent_id` と `cloid` の両方保持
- reconciliation は `cloid` 主、reporting は `intent_id` 主

**状態**  
承認
